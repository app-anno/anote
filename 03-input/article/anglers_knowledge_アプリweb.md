テキストの上下中央揃えがOSで揃わない問題
問題の概要
→ iOSではだいたい大丈夫だが、Androidではセンタリングがズレる ★
→ 原因 → フォントの高さ基準がOSで異なる
フォント高さの2つの基準
 フォントには2種類の高さ基準がある。赤い線（Ascent/Descent）はフォントギリギリの上下端、青い線（Top/Bottom）はマージンを含めた設定位置。
① Ascent / Descent（赤い線）→ フォント本体のギリギリの境界
② Top / Bottom（青い線）→ マージンを含む外側の境界
→ iOS → Ascent / Descent を採用（ギリギリ基準）
→ Android → Top / Bottom を採用（デフォルト）★
 TopとBottomのスペースは非対称で、Topが広くBottomが狭い。そのままレンダリングするとフォントが下寄りに表示されてしまう。
解決方法
Android側の調整
① includeFontPadding を false に設定 → iOSと同じ基準に揃える
② lineHeight を fontHeight の 1.2〜1.5倍に設定 → フォント途切れ防止
《例》フォントサイズによって倍率が変わる（Material Design ガイドライン参照）
③ textAlignVertical: 'center' を使用 → 上下方向のセンタリング
iOS側の調整
→ デフォルトでほぼ問題なし
→ flexbox で中央揃えすれば高さが揃う
→ カスタムフォント使用時の微調整
① fontSize と lineHeight を同じ値に設定
② フレックスボックスでセンタリング
 iOSにはtextAlignVerticalがないため、フォント上下の隙間をほぼゼロにしてからflexboxでセンタリングする方法を取る。
 テキストコンポーネントをラップした共通コンポーネントを作成し、プロパティでセンタリングスタイルを適用する設計がおすすめ。
実践的な使い方
 Android: includeFontPadding={false} + lineHeight調整 + textAlignVertical='center'
 iOS: fontSize = lineHeight + flexbox センタリング
 共通ラッパーコンポーネントを作成してOS差分を吸収する

依存ライブラリを減らすために車輪を再開発してもいい
背景と課題
→ 秘伝の継ぎ足しプロジェクトのメンテナンス時に依存地獄に遭遇
→ React v19 対応が必要 → React Recoil が消滅 ★
→ ライブラリアップデートがスパゲッティ状態で連鎖
→ React Native 0.7x 時代のアーキテクチャ変更への対応も必要
精査結果
・パッケージ削除 → 13個
・お手製（車輪の再開発）に置き換え → 6個
具体例：タイムゾーンライブラリの置き換え
→ 従来：外部ライブラリでタイムゾーンオブジェクトを取得
→ 実際の使用箇所 → たった1箇所

 ネイティブAPIで代替可能なら、ライブラリのアップデート管理から解放される。

→ Claude、Codex、Gemini などで一発生成 → 一気に置き換え可能
→ UIパーツも同様にCSSモジュールへ変換可能

② 最終責任は自分 → 出力コードは必ず自分で確認

React Native New Architecture 移行実践報告
New Architecture の概要
→ JSI（JavaScript Interface）→ ネイティブとJSの直接通信
→ Turbo Modules → 遅延ロード可能なネイティブモジュール
→ Fabric → 新しいレンダリングシステム
→ Codegen → TypeScript/Flow からネイティブコード自動生成
iOS側の実装
→ Podfile に設定追加
→ TurboModules の記述が必要
Android側の実装（より複雑）★
① build.gradle に New Architecture 有効判定処理を追加
② ソースコードパスの指定
③ Bridge Package の作成
④ Bridge Module に処理を実装
⑤ @ReactMethod → Override に変更
⑥ 継承クラスが Codegen で自動生成されたクラスに変更
開発中に遭遇した課題
Codegen の理解が困難 ★
→ 生成コードをSDKに入れるべきか不明
→ 結論：アプリビルド時に一緒に生成されるので入れなくてOK
日本語の移行事例がほぼ皆無 ★
→ Firebase や Braze のソースコードを参考に
→ 後方互換の実装例が特に見つからない → Gemini に相談
ドキュメント・サンプルが消えていく問題 ★★
 4月から開発開始。サンプルアプリは古くて動作せず、7月1日にリポジトリ自体が削除された。
《例》globalThis.__turboModuleProxy が 2024年のPRで削除済み → Geminiも古い情報を提示
型変換の罠
→ TypeScript の number 型 → Java の Double 型に変換
→ ドキュメントの最下部に記載 → 探すのに時間がかかる
後方互換の実装方法
→ index.js で New Architecture かどうかを判定して分岐
→ 2つの関数を用意（レガシー用・New Architecture用）
→ Spec定義のメソッドで直通させる
 約2ヶ月で開発完了。Gemini のおかげで乗り越えられた。
 New Architecture 対応は闇が深い。過去のライブラリ実装を参考にするしかないが、古すぎると参考にならないジレンマ。

《例》Music アプリの再生バー、Podcast アプリのミニプレーヤー
